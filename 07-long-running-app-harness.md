# 长时间应用开发的 Harness 设计

> 目标：把 Anthropic Engineering 的长时间应用开发 harness 思路，转译成 Claude Code 用户可以直接落地的项目实践。

本文参考 Anthropic Engineering 文章 [Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps)，但不逐段翻译原文，而是结合 Claude Code 的真实使用方式，整理成一篇面向实践的 guide。

如果你还没有读过方法论总纲，建议先读 [Claude Code 的 Harness Engineering 总纲](./06-harness-engineering-guide.md)。

---

## 这篇文章解决什么问题

短任务里，Claude Code 的协作方式通常很简单：

1. 读项目
2. 修改文件
3. 跑检查
4. 修到通过

但当任务变成长时间应用开发时，问题会明显复杂很多。比如：

- 从一句产品想法开始，生成完整应用
- 连续几个小时实现多个功能模块
- 在前端设计、后端逻辑、测试和修复之间来回迭代
- 让不同 agent 分别负责规划、实现和评估
- 在上下文增长后仍然保持任务方向稳定

这时，单纯让一个 agent “一直做下去”并不可靠。真正需要设计的是一套 harness：

> 用结构化分工、上下文交接、外部评估和验证回路，让 Claude Code 能稳定完成长时间、多阶段的软件任务。

---

## 原文的核心观点

Anthropic 这篇文章围绕一个问题展开：如何让 Claude 在长时间应用开发中走得更远。

它总结了几个关键经验：

1. harness 设计会显著影响 agentic coding 的表现
2. 长任务容易出现上下文衰退和过早收尾
3. 任务拆分和结构化交接能提升稳定性
4. 自我评估不可靠，需要把“做事的 agent”和“评价的 agent”分开
5. 前端设计这类主观任务，也可以通过评分标准变得更可评估
6. 多 agent 不应只是堆角色，而要对应真实 failure mode
7. 模型升级后，要重新审视 harness，删除不再必要的脚手架
8. 最好的 harness 不是最复杂的，而是刚好补上模型短板的那一个

这与本 guide 前面反复强调的原则一致：

> 同一个问题出现两次，就不要再靠人提醒，应该把它写进 harness。

## 长任务最常见的两个失败模式

### 1. 上下文越来越满，任务越来越散

长时间开发中，agent 会积累大量上下文：

- 需求讨论
- 已读文件
- 修改历史
- 报错输出
- 中间决策
- 未完成任务

上下文越来越长以后，模型可能开始遗忘早期约束，也可能逐渐偏离原始目标。

原文提到一个现象：有些模型在接近上下文限制时，会出现类似“上下文焦虑”的行为，提前把任务收尾，即使实际还没有完成。

在 Claude Code 里，你会看到类似表现：

- 还没跑完整检查就说完成了
- 只修了表层问题，没有继续追根因
- 忘记最初要求的边界条件
- 越到后面越倾向于总结，而不是继续推进

### 2. 自我评估过于宽松

让同一个 agent 既写代码又评价自己的代码，往往不够可靠。

它可能会：

- 发现问题后把问题说小
- 对 UI 质量给出过高评价
- 只测试 happy path
- 没有主动探索边界场景
- 把“看起来可用”当成“真的完成”

这也是为什么在真实项目里，review、QA、测试、CI 都不是“写代码的人自我声明一下”就结束。

Claude Code harness 也应该遵循同样的工程常识：

> 生成和评估要分离。

## 三角色结构：Planner / Generator / Evaluator

原文最终形成的是一个三 agent 结构：

| 角色 | 职责 | Claude Code 中的落点 |
| --- | --- | --- |
| Planner | 把简短想法扩展成产品规格和高层设计 | Ask / Plan、专用 planning subagent、`docs/spec.md` |
| Generator | 根据规格实现应用 | 主 Claude Code 会话或实现型 subagent |
| Evaluator | 独立检查功能、设计、交互和完成度 | review subagent、test-runner、Playwright MCP、检查脚本 |

这三个角色不是为了“看起来像多 agent”，而是分别解决不同问题：

- Planner 解决“需求不完整”
- Generator 解决“持续实现”
- Evaluator 解决“自我评估不可靠”

如果你的项目还很小，不一定要真的配置三个 subagent。你可以先用同一个 Claude Code 会话手动模拟这三个阶段：

```text
先不要写代码。请把这个需求扩展成一份产品规格，只写目标、用户流程、核心功能和验收标准，不要写具体实现细节。
```

```text
根据刚才的规格，先实现第一阶段功能。每完成一个功能点后运行对应检查，不要一次性改太大范围。
```

```text
现在换成审查者视角，不要替刚才的实现辩护。请根据验收标准逐项检查，列出必须修复的问题。
```

这已经是最小版本的三角色 harness。

## Planner：不要过早规定实现细节

原文里，Planner 的任务不是写一份超细的技术方案，而是把 1-4 句话的 prompt 扩展成完整产品规格。

关键点是：Planner 应该关注产品上下文和高层技术设计，而不是提前规定所有底层实现细节。

原因很简单：如果 Planner 在一开始写错了细节，后续 Generator 可能会忠实执行错误设计，导致错误被放大。

更好的规格结构是：

```md
# 产品规格

## 目标

这个应用要解决什么问题。

## 用户角色

谁会使用它，他们分别要完成什么任务。

## 核心流程

用户从进入应用到完成关键任务的路径。

## 功能范围

### 必须实现

- ...

### 可以延后

- ...

## 验收标准

- 用户可以完成 A
- 用户可以完成 B
- 错误状态 C 有明确反馈
- 核心流程有自动化或手动验证方式

## 非目标

- 本轮不做 ...
```

在 Claude Code 中，建议把 Planner 的输出写进真实文件，例如：

```text
docs/spec.md
```

这样后续实现、评估和复盘都有一个稳定锚点。

## Generator：按可交付块推进，而不是一口气做完

长任务最怕“一次性实现整个应用”。

更稳定的方式是把实现拆成可交付块：

1. 项目骨架
2. 核心数据模型
3. 第一个可用流程
4. 关键交互
5. 错误处理
6. 测试与验证
7. UI polish

每一块都应该有明确的完成条件。

示例提示词：

```text
按照 docs/spec.md 实现第一阶段：只完成核心数据模型和最小可用主流程。
完成后运行类型检查和最相关的测试。不要开始第二阶段。
```

当任务很长时，不要让 Claude Code 只靠聊天历史记住进度。让它维护结构化状态文件：

```md
# 开发状态

## 已完成

- [x] 项目骨架
- [x] 核心数据模型

## 当前阶段

实现最小可用主流程。

## 待处理

- [ ] 错误状态
- [ ] 表单校验
- [ ] Playwright 验证

## 已知问题

- ...

## 下一步

- ...
```

可以放在：

```text
docs/agent-state.md
```

这个文件就是长任务的 handoff artifact。

## Evaluator：让另一个角色负责挑刺

Evaluator 的任务不是“看起来不错就通过”，而是主动找问题。

一个好的 Evaluator prompt 应该明确要求它：

- 不要替实现辩护
- 不要只检查 happy path
- 按验收标准逐项核对
- 区分 blocker、major、minor
- 明确哪些问题必须修复后才能完成
- 如果是 UI，要实际打开页面检查

示例：

```text
你现在是独立 QA，不是实现者。
请根据 docs/spec.md 和当前应用行为做验收。

要求：
1. 不要评价“整体不错”，只列证据充分的问题。
2. 至少覆盖一个正常路径、一个错误路径、一个边界路径。
3. 把问题分为 blocker / major / minor。
4. blocker 未清零前，不允许判断任务完成。
5. 如果需要浏览器验证，请实际打开页面操作。
```

如果项目是 Web 应用，Evaluator 最好结合浏览器或 Playwright：

```text
启动开发服务器，打开核心页面，按用户流程实际操作一遍。
不要只读代码判断 UI 是否可用。
```

这和本 guide 前面强调的原则一致：

> 类型检查和测试能验证代码正确性，但不能替代真实功能体验验证。

## 评分标准：把主观质量变成可讨论对象

前端设计、交互体验、产品完整度并不是完全客观的。

但这不意味着它们无法进入 harness。

原文的做法是：给 Generator 和 Evaluator 同一组评分标准，让“好不好”变成更具体的问题。

你可以为 UI 任务写这样的标准：

| 维度 | 评价问题 |
| --- | --- |
| 功能完整度 | 用户能否完成核心任务，而不只是看到静态界面 |
| 交互深度 | 按钮、表单、拖拽、筛选等是否真的可用 |
| 信息架构 | 页面层级是否清楚，用户是否知道下一步做什么 |
| 视觉质量 | 是否有一致的间距、字体、颜色和组件风格 |
| 原创性 | 是否只是常见模板感，还是有贴合产品的设计选择 |
| 错误状态 | 加载、空状态、失败状态是否可理解 |

把这些写进 `docs/spec.md` 或 `CLAUDE.md`，Evaluator 才有明确依据。

## 上下文重置与交接文件

原文区分了两种长任务续航方式：

- compaction：压缩当前对话，让同一个 agent 继续
- reset：清空上下文，用结构化交接文件启动新 agent

compaction 保留连续性，但旧上下文的噪音和心理惯性也可能留下。

reset 给新 agent 一个干净起点，但要求 handoff artifact 足够完整。

在 Claude Code 日常使用里，不一定每次都要主动 reset。更实用的策略是：

| 场景 | 建议 |
| --- | --- |
| 任务还短，方向稳定 | 继续当前会话 |
| 上下文很长，但状态清晰 | 先整理 `docs/agent-state.md`，再继续 |
| Claude 开始反复总结、跑偏或漏步骤 | 停下来写 handoff，再开新会话 |
| 任务跨天或跨多人 | 必须写结构化交接文件 |

交接文件至少包含：

```md
# Handoff

## 原始目标

...

## 当前代码状态

...

## 已完成

...

## 未完成

...

## 必须遵守的约束

...

## 最近一次验证结果

...

## 下一步建议

...
```

关键是：不要让下一轮 Claude 只能从聊天记录里猜状态。

## 最小可行长任务 harness

如果你想在自己的项目里落地，可以先从这个最小版本开始：

```text
your-project/
├── CLAUDE.md
├── docs/
│   ├── spec.md
│   ├── agent-state.md
│   └── handoff.md
├── scripts/
│   └── ai/
│       └── run-checks.sh
└── .claude/
    ├── settings.json
    └── agents/
        └── code-reviewer.md
```

对应关系：

| 文件 | 作用 |
| --- | --- |
| `CLAUDE.md` | 项目规则、边界和常用命令 |
| `docs/spec.md` | Planner 输出，定义目标和验收标准 |
| `docs/agent-state.md` | Generator 持续维护的状态 |
| `docs/handoff.md` | reset 或跨会话交接用 |
| `scripts/ai/run-checks.sh` | deterministic 验证入口 |
| `.claude/agents/code-reviewer.md` | 独立 Evaluator 角色 |

如果还没有准备好 subagent，也可以先只建前三个 docs 文件和检查脚本。

## 一套可直接使用的长任务流程

下面是一套可以直接复制到 Claude Code 里使用的流程。

### 第一步：生成规格

```text
先不要写代码。
请根据我的一句话需求生成 docs/spec.md。
要求只写产品目标、用户流程、功能范围、验收标准和非目标。
不要过早规定具体技术实现。
```

### 第二步：拆分阶段

```text
读取 docs/spec.md，把实现拆成 4-6 个可交付阶段，写入 docs/agent-state.md。
每个阶段都要有完成条件和验证方式。
```

### 第三步：执行一个阶段

```text
只执行 docs/agent-state.md 里的当前阶段。
完成后更新 docs/agent-state.md，并运行相关检查。
不要自动进入下一阶段。
```

### 第四步：独立评估

```text
现在切换到独立 Evaluator 视角。
根据 docs/spec.md 和实际运行结果验收当前阶段。
请列出 blocker / major / minor，并给出复现步骤。
```

### 第五步：修复并复验

```text
只修复 Evaluator 标记为 blocker 和 major 的问题。
修复后重新运行检查，并更新 docs/agent-state.md。
```

### 第六步：准备交接

```text
把当前状态整理到 docs/handoff.md。
内容包括原始目标、已完成、未完成、验证结果、已知问题和下一步。
```

这套流程的重点不是“让 Claude 更忙”，而是让 Claude 的工作留下可验证、可接手、可复盘的轨迹。

## 什么时候需要多 agent

不要为了多 agent 而多 agent。

只有当你看到明确 failure mode 时，才值得拆角色。

| 现象 | 可以引入的角色 |
| --- | --- |
| 需求总是不清楚 | Planner |
| 实现经常偏离目标 | Spec reviewer |
| 代码写完但质量不稳 | Code reviewer |
| UI 看起来能用但交互很浅 | UX evaluator |
| 测试经常漏跑 | Test runner |
| 长任务后期状态混乱 | Handoff summarizer |

每个 agent 都应该回答一个问题：

> 它补的是哪个具体短板？

如果回答不出来，就先不要加。

## 模型升级后，重新审视 harness

原文最后有一个很重要的提醒：模型能力会变，harness 也要跟着变。

一个在旧模型上必要的脚手架，到了新模型上可能会变成负担。

所以，每次你升级模型或明显感觉模型能力变化时，都应该问：

1. 这个 hook 还必要吗？
2. 这个 subagent 还真的提升质量吗？
3. 这个 handoff 文件是不是太重了？
4. 哪些检查可以自动化，哪些还需要人工判断？
5. 有没有新的能力可以让 harness 更简单？

好的 harness 不是越复杂越好，而是：

> 每个组件都仍然 load-bearing。

如果一个组件不再承担关键作用，就应该删掉或简化。

---

## 注意事项

### 不要把 Planner 变成架构独裁者

Planner 应该定义目标和验收标准，而不是提前规定所有实现细节。

### 不要相信自我验收

让实现者自己说“已经完成”是不够的。至少要有检查脚本、测试、review 或浏览器验证之一。

### 不要让状态只存在聊天里

长任务一定要有文件化状态。否则跨会话、跨天、跨人都会丢信息。

### 不要一次上满复杂系统

先用文档和检查脚本搭最小 harness。只有当问题反复出现时，再引入 subagent、MCP、hooks 或更复杂的编排。

---

## 总结

长时间应用开发的关键，不是让 Claude Code 单次变得更“聪明”，而是给它一个能持续工作的环境：

- Planner 把模糊需求变成规格
- Generator 按可交付块实现
- Evaluator 独立挑刺
- 状态文件承接长上下文
- 检查脚本和真实运行结果提供 back-pressure

这就是 Harness Engineering 在长任务里的具体形态。

当任务短时，直接协作就够了；当任务变长时，就要把协作方式升级为系统。
