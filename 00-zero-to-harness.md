# Claude Code：从 Zero 到 Harness

> 目标：帮第一次系统使用 Claude Code 的读者，走完从“会用 Claude Code”到“能搭出最小可行 harness”的认知路径。

## 这篇文章解决什么问题

很多人第一次接触 Claude Code，会先感受到它的强：

- 能读整个项目
- 能解释多文件关系
- 能直接改代码
- 能运行测试并根据反馈继续修改

但只要开始在真实项目里持续使用，很快就会遇到第二层问题：

- 为什么同样一句话，这次改得好，下次却跑偏？
- 为什么它知道怎么写代码，却不知道我们团队怎么做事？
- 为什么它能生成很多东西，却不一定稳定地按项目要求收尾？

这时你会发现，真正的问题已经不是“会不会用 Claude Code”，而是：

> 能不能把 Claude Code 变成一个长期稳定、可复用、可验证的工程协作系统。

这就是 harness 要解决的事。

所以，这篇文章不是安装手册，也不是 CLI 参数大全，而是一条面向新手的路线图：

1. 先把 Claude Code 用起来
2. 再把 Claude Code 放进项目上下文里
3. 最后把反复出现的问题沉淀成 harness

---

## 一、先建立一个正确直觉：Claude Code 不是聊天框

很多人第一次接触 Claude Code，会下意识把它理解成“能写代码的 Claude 聊天窗口”。

这个理解不完全错，但不够准确。

Claude Code 和网页版 Claude 的关键差异，不是回答风格，而是工作方式：

| 维度 | 网页版 Claude | Claude Code |
| --- | --- | --- |
| 主要交互 | 聊天 | 面向项目的工程协作 |
| 上下文来源 | 你手动粘贴的内容 | 当前代码仓库、文件、命令、配置 |
| 执行动作 | 输出文本 | 读文件、分析代码、修改代码、执行检查 |
| 约束方式 | 主要靠对话提醒 | 可以固化为项目级规则与权限边界 |

换句话说：

- 在网页版里，你是在“向 Claude 描述项目”
- 在 Claude Code 里，Claude 是“进入项目本身工作”

这也是为什么 Claude Code 的价值，不止是“帮你生成代码”，而是“在真实仓库里参与开发”。

### 为什么这件事很重要

如果你把 Claude Code 当聊天工具来用，你会更关注：

- 这一轮回答聪不聪明
- 这一段代码写得漂不漂亮
- 这次改动是不是立刻可用

但如果你把 Claude Code 当工程协作系统来用，你就会开始关心：

- 它是否理解项目结构
- 它是否遵守团队规则
- 它是否知道什么能做、什么不能做
- 它是否在“完成任务”前经过验证

从这里开始，你已经在接近 harness 思维了。

## 二、先把 Claude Code 用起来，而不是一上来就搭系统

很多新手一听到 harness、hooks、MCP、subagents，就会误以为：

> 想把 Claude Code 用好，必须先搭一整套复杂系统。

其实不是。

正确顺序永远是：

1. 先完成几次真实的小任务
2. 观察 Claude Code 在项目里怎么工作
3. 再把反复出现的问题沉淀成规则和机制

也就是说，**先会用，再固化。**

### 从最小任务开始

第一次上手时，不要直接给 Claude Code 一个巨大任务。更好的做法是让它先完成几类小而明确的工作：

#### 1. 解释代码

例如：

```text
解释 auth.ts 里的登录流程，用新手能理解的方式说明。
```

这个阶段最重要的不是“让它改代码”，而是确认它是否真的能读懂你的项目上下文。

#### 2. 分析问题

例如：

```text
这个接口为什么一直返回 400？先不要修改代码，帮我定位原因。
```

这里你会开始感受到 Claude Code 和普通聊天模型的差别：它不是只围绕一段粘贴内容给建议，而是会去看相关文件、依赖关系和上下文。

#### 3. 做小范围修改

例如：

```text
在不改变现有行为的前提下，帮我提升这个函数的可读性。
```

当你做过几次这样的任务，就会逐渐形成一个判断标准：

- 什么样的问题适合直接问 Claude Code
- 什么样的任务需要先解释、再规划、再执行
- Claude Code 在你的项目里最容易在哪些地方出错

这些观察，会直接变成后面搭 harness 的依据。

### 学会 Ask / Plan / Edit 的边界

在真正进入 harness 之前，你最好先建立 Claude Code 的三种基本使用姿势：

#### Ask：先理解，不动手

适合这些场景：

- 刚接手陌生代码
- 还不确定 bug 原因
- 想先知道改动影响范围

Ask 的价值在于：**先把问题看清楚，再决定要不要动代码。**

#### Plan：先对齐，再执行

适合这些场景：

- 涉及多个文件
- 涉及模块边界或架构调整
- 你还没有完全想清楚怎么改

Plan 的价值在于：**把“先写代码”改成“先达成共识”。**

#### Edit：明确之后再动手

适合这些场景：

- 目标已经明确
- 改动范围可控
- 你知道改什么，也知道怎么验

Edit 的价值在于：**在明确边界后精确执行。**

如果只记住一句话，那就是：

> 不确定时先 Ask；复杂改动先 Plan；目标清楚后再 Edit。

这套节奏本身，就是你未来 harness 的基础工作流。

## 三、从一次性使用到项目化使用

当你完成了几次真实任务后，会进入第二个阶段：

> 问题已经不是“Claude 会不会写”，而是“Claude 知不知道这个项目该怎么做事”。

这时，项目上下文就变得比单轮 prompt 更重要。

### `/init` 的意义，不是生成文件，而是建立项目入口

很多人第一次运行 `/init`，会觉得它只是“帮我生成一个 `CLAUDE.md`”。

但从项目化使用的角度看，`/init` 更重要的意义是：

- 让 Claude Code 扫描当前仓库
- 识别技术栈、目录、命令和已有约束
- 为项目建立一个长期可复用的上下文入口

你可以把它理解成：

> 给 Claude 写一份项目欢迎手册，而不是每次开场都临时口头交代。

### `CLAUDE.md` 是项目级记忆，不是万能提示词

`CLAUDE.md` 很容易被误解成“把所有要求都塞进去的地方”。

更准确的理解是：

> `CLAUDE.md` 负责告诉 Claude：这个项目是什么、有哪些边界、应该怎么验证，而不是替代所有工程工具。

一份好的 `CLAUDE.md`，通常至少包含三类信息：

- **WHAT**：项目是什么，目录怎么分，关键模块在哪
- **WHY**：哪些边界和约束是重要的，为什么不能乱改
- **HOW**：开发命令、验证方法、协作流程是什么

一个最小示例大概像这样：

```markdown
# Project Memory

## Mission
- 这个项目要解决什么问题
- 哪些事情不是目标

## Architecture
- 关键目录如何分工
- 哪些边界不能跨

## Guardrails
- 哪些文件不要直接改
- 哪些约束必须遵守

## Commands
- test: `npm test`
- lint: `npm run lint`
- typecheck: `npm run typecheck`

## Workflow
- 复杂改动先给方案
- 完成前必须运行检查
```

这里的关键，不是写得多，而是写得稳。

### 不要把 `CLAUDE.md` 当 linter、测试框架或代码审查替代品

这是新手最常见的误区之一。

比如你可以在 `CLAUDE.md` 里写：

- 优先补测试再改实现
- 不要跨模块直接调用内部实现
- 涉及复杂改动先给方案

但你不应该指望它替代这些东西：

- linter
- typecheck
- 测试
- 权限控制
- 自动化校验

原因很简单：

- 提示词是概率性的
- 工具检查是确定性的

所以，项目化使用 Claude Code 的第一步，不是继续堆提示词，而是开始区分：

- 哪些规则只需要告诉 Claude
- 哪些规则必须被工具和流程强制执行

这个分界线，就是 harness 的入口。

## 四、什么是 Harness

如果用一句最短的话来解释：

> Harness 就是把 Claude Code 的行为，从“临场发挥”变成“有上下文、有边界、有验证的稳定协作”。

它不是另起一套 agent framework，也不是一堆花哨自动化。

它更像是你为 Claude Code 搭出来的一套控制系统。

### Claude Code 自带的 harness 原语

Claude Code 本身已经提供了很多可直接利用的原语：

- `CLAUDE.md`
- `.claude/settings.json`
- hooks
- slash commands
- subagents
- MCP

所以大多数团队真正要做的，并不是“从零造一个 AI 平台”，而是：

> 在 Claude Code 已有原语之上，补齐你自己项目的上下文、权限、工具和验证机制。

### 用五层结构理解 harness

最适合新手入门的方式，不是记概念，而是把 harness 看成五层结构：

| 层 | 目标 | 典型载体 |
| --- | --- | --- |
| Memory | 让它知道“我们这里怎么做事” | `CLAUDE.md` |
| Permissions | 让它知道“什么能做，什么不能做” | `.claude/settings.json` |
| Tools | 让它能接触需要的工具和检查命令 | scripts、commands、MCP |
| Delegation | 让复杂任务按角色拆分 | subagents |
| Back-pressure | 让“完成”必须经过验证 | hooks + lint/test/typecheck/review |

这五层并不要求你一次全做。

它真正要表达的是：

- Claude 不只需要“会做事”
- Claude 还需要“知道按什么规则做事”
- 更需要“做完之后怎么证明自己真的做完了”

### Harness 的本质：把重复问题写进系统

最值得记住的一条原则是：

> 同一个问题出现两次，就不要继续靠人提醒，而应该把它写进 harness。

比如：

- Claude 总是忘记先跑测试
- Claude 总会去碰敏感目录
- Claude 在复杂任务里容易漏掉 review
- Claude 对某些模块边界理解不稳定

第一次，你可以手动提醒。
第二次，就该考虑把它固化进：

- `CLAUDE.md`
- settings 权限
- hook
- check script
- subagent
- slash command

这就是从“使用 AI”走向“工程化使用 AI”的关键一步。

## 五、一个最小可行 Harness 长什么样

新手最容易犯的另一个错误，是一开始就想把 hooks、MCP、skills、commands、agents 全部接上。

其实第一版 harness 应该尽量小。

目标不是“功能齐全”，而是：

> 先让 Claude 在你的项目里稳定、不跑偏、能收尾。

对于大多数项目，一个最小可行 harness 通常只需要下面五样东西。

### 1. 一个精简的 `CLAUDE.md`

它解决的问题是：Claude 进入项目时，至少知道这是个什么项目、有哪些边界、常用命令是什么。

你不需要写成长篇手册，更不需要把所有编码规范都写进去。

优先写这些：

- 项目目标和非目标
- 关键目录和模块边界
- 测试 / lint / typecheck 命令
- 复杂改动前的工作流要求

### 2. 一个统一的检查脚本

如果每次都靠 Claude 自己猜该跑什么命令，它迟早会猜错。

所以最好给它一个固定入口，例如：

```bash
#!/usr/bin/env bash
set -euo pipefail

npm run lint
npm run typecheck
npm test
```

这样做的好处不是“少写几个命令”，而是把“完成前该怎么验证”写成一个明确的、可复用的动作。

### 3. 一层基本权限边界

不是所有文件都适合默认暴露给 Claude。

例如：

- `.env`
- `secrets/`
- 生产数据目录
- 高风险命令

这些东西不应该主要靠口头提醒，而应该放进权限配置里。

一个最小示例可以是：

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)",
      "Read(./prod-data/**)"
    ],
    "ask": [
      "Bash(git push:*)",
      "Bash(rm:*)"
    ]
  }
}
```

### 4. 一个 Stop hook

这一步特别关键。

很多团队的问题不是 Claude 不会生成代码，而是它太容易在“看起来差不多了”的时候停下来。

Stop hook 的意义，就是给“完成”加一个硬门槛。

例如：

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "./scripts/ai/run-checks.sh"
          }
        ]
      }
    ]
  }
}
```

这会把“做完”从自然语言状态，变成一个需要经过检查的状态。

### 5. 一个只负责 review 的子代理

第一批子代理不需要多。

最值得先加的，通常是一个只读 reviewer，而不是一个“大而全的万能代理”。

例如它只负责检查：

- 有没有行为回归
- 有没有漏测试
- 有没有跨模块越界
- 有没有明显的可维护性问题

这样做的价值在于：

- 主对话专注实现
- reviewer 专注挑错
- 结果更稳定，也更省上下文

### 最小可行，不等于简陋

很多人会把“最小可行 harness”理解成“先随便搭一个”。

其实更准确的理解是：

- 只保留最有杠杆的部件
- 先解决最常重复的问题
- 先保证稳定性，再追求扩展性

对于新手来说，这比一开始铺满高级功能有用得多。

## 六、什么时候再加高级能力

当最小可行 harness 已经跑顺，你才需要考虑更强的扩展能力。

一个比较稳的顺序通常是这样：

### 第一步：先把项目记忆和检查跑顺

也就是先稳定这几样：

- `CLAUDE.md`
- 基础权限
- `run-checks.sh`
- Stop hook

如果这一步还没稳，就不要急着上更多能力。

### 第二步：再加角色分工

当你发现主对话里经常混杂这些任务时：

- 搜索资料
- 代码审查
- 跑测试
- 排查问题

就可以开始考虑加 subagents。

但原则仍然是：

> 一开始只为明确角色建代理，不为“看起来高级”建代理。

### 第三步：把高频流程做成 commands 或固定脚本

当某类流程已经反复出现，例如：

- 实现后自动 review
- 修 bug 前先收集上下文
- 提交前统一检查

就可以考虑把它们沉淀成命令或脚本，而不是每次重新描述。

### 第四步：最后按需接入 MCP

MCP 很强，但它不是新手第一阶段的必需品。

只有当你真的需要 Claude 连接项目之外的系统时，再考虑引入，例如：

- GitHub / issue 系统
- Sentry / 日志平台
- 数据库只读查询
- 设计稿或外部知识库

一个很实用的判断标准是：

> 如果问题主要发生在当前仓库内部，先别急着上 MCP。

## 七、常见误区

### 误区 1：把 `CLAUDE.md` 当万能药

`CLAUDE.md` 很重要，但它只能解决“告诉 Claude 什么是对的”这类问题。

它不能单独解决：

- 强制验证
- 权限控制
- 自动收尾
- 外部系统集成

所以它是 harness 的起点，不是 harness 的全部。

### 误区 2：把 harness 理解成“写很多提示词”

harness 当然包含提示，但它的本质不是“提示词工程堆料”，而是：

- 版本化上下文
- 明确权限边界
- 稳定工具入口
- 角色分工
- 可验证完成状态

如果缺少这些，提示词写再多，也还是容易漂。

### 误区 3：一开始就堆太多工具

hooks、skills、subagents、MCP、commands 都有价值，但它们不是越多越强。

过早堆工具的典型后果是：

- 配置越来越多，但团队没人真正常用
- 问题来源不清楚，不知道是哪层在失效
- 新人一上来就被复杂度劝退

最好的策略永远是：

> 先解决最常见、最痛的重复问题，再扩展下一层。

### 误区 4：只关注生成，不关注收尾

Claude Code 很擅长开始做事，但工程场景里更难的是“收尾”。

真正成熟的使用方式，不是让它多快产出第一版，而是让它在结束前完成这些事：

- 跑检查
- 暴露失败
- 接受 review
- 回到项目边界内修正问题

也就是说，重点不只是生成能力，而是 back-pressure。

## 八、推荐的起步顺序

如果你今天就想从 zero 开始迈向 harness，可以按下面顺序走。

### 第 1 步：先做三到五次真实小任务

先别追求系统化，重点是观察 Claude Code 在你项目里的真实表现。

### 第 2 步：用 `/init` 建立项目入口

拿到第一版 `CLAUDE.md`，然后人工收敛，让它更贴近真实项目，而不是保留一堆泛化描述。

### 第 3 步：把常用检查统一成一个脚本

让“怎么验收”变成固定入口，而不是每次临时决定。

### 第 4 步：补一层最小权限和 Stop hook

把最容易出问题的敏感区和收尾流程先保护起来。

### 第 5 步：增加一个 reviewer 子代理

让实现和审查开始分工。

### 第 6 步：只有在需要时，再接着扩展

例如 commands、更多 agents、MCP、外部系统集成。

这套顺序的核心思想很简单：

> 先让 Claude Code 在项目里站稳，再让它变强。
